## Wave-Based Memory - Wie es funktioniert
### Normale Embeddings (Real-valued)
"happy"     → [0.8, 0.3, 0.5, ...]

"not happy" → [0.75, 0.35, 0.48, ...]  ← Fast identisch!

Cosine Similarity ≈ 0.95  ← Problem: Negation geht verloren
### Wave-Based (Complex-valued)
Jedes Item ist eine **Welle** mit Amplitude UND Phase:
Darstellung: z = A · e^(iφ)

"happy"     → Amplitude=0.8, Phase=0°      → 0.8 + 0i

"not happy" → Amplitude=0.8, Phase=180°    → -0.8 + 0i  ← Entgegengesetzt!
### Retrieval durch Interferenz
Query-Welle trifft auf gespeicherte Wellen:

┌─────────────────────────────────────────────────┐

│                                                 │

│   Query: "happy"  (Phase 0°)                    │

│        ∿∿∿∿∿∿∿∿∿∿                               │

│                                                 │

│   Doc1: "happy"   (Phase 0°)                    │

│        ∿∿∿∿∿∿∿∿∿∿                               │

│        ══════════  ← Konstruktiv = MATCH        │

│                                                 │

│   Doc2: "not happy" (Phase 180°)                │

│        ∿∿∿∿∿∿∿∿∿∿  (invertiert)                 │

│        ──────────  ← Destruktiv = ANTI-MATCH    │

│                                                 │

└─────────────────────────────────────────────────┘
### Mathematisch
# Normal (Cosine)

similarity = dot(query, doc) / (norm(query) * norm(doc))

# Wave (Resonance)

# query und doc sind komplexe Vektoren

resonance = |sum(query * conj(doc))|² / normalization

# Die Phase macht den Unterschied:

# e^(i·0°) * e^(-i·0°)   = 1      (Match)

# e^(i·0°) * e^(-i·180°) = -1     (Anti-Match)

class PhasePredictor(nn.Module):
    """Lernt Phase aus semantischen Gegensätzen"""
    
    def __init__(self, input_dim=768, hidden_dim=256):
        super().__init__()
        self.net = nn.Sequential(
            nn.Linear(input_dim, hidden_dim),
            nn.GELU(),
            nn.Linear(hidden_dim, hidden_dim),
            nn.GELU(),
            nn.Linear(hidden_dim, 1),  # Globale Phase oder per-dim
            nn.Sigmoid()  # Output 0-1, dann * 2π
        )
    
    def forward(self, embedding):
        return self.net(embedding) * 2 * math.pi

# Training mit Contrastive Loss
def contrastive_phase_loss(anchor_phase, positive_phase, negative_phase):
    """
    Positive Paare: ähnliche Phase (konstruktive Interferenz)
    Negative Paare: entgegengesetzte Phase (destruktive Interferenz)
    """
    # Phasen-Differenz sollte klein sein für positive Paare
    pos_diff = torch.abs(torch.cos(anchor_phase - positive_phase) - 1)
    
    # Phasen-Differenz sollte ~π sein für negative Paare  
    neg_diff = torch.abs(torch.cos(anchor_phase - negative_phase) + 1)
    
    return pos_diff.mean() + neg_diff.mean()
#+END_SRC

class WaveEmbedder:
    def __init__(self, base_model_path, phase_model_path):
        # Frozen base embedder (dein existierendes Modell)
        self.embedder = SentenceTransformer(base_model_path)
        self.embedder.eval()
        
        # Trainierter Phase Predictor
        self.phase_predictor = PhasePredictor()
        self.phase_predictor.load_state_dict(torch.load(phase_model_path))
        self.phase_predictor.eval()
    
    def encode(self, text: str) -> np.ndarray:
        with torch.no_grad():
            # Amplitude aus Base-Embedder
            amplitude = self.embedder.encode(text, normalize_embeddings=True)
            amplitude_tensor = torch.tensor(amplitude)
            
            # Phase aus Predictor
            phase = self.phase_predictor(amplitude_tensor)
            
            # Komplexe Welle konstruieren
            # z = A * e^(iφ) = A * (cos(φ) + i*sin(φ))
            wave = amplitude * np.exp(1j * phase.numpy())
            
        return wave  # Shape: [768] complex64
    
    def resonance_score(self, query_wave, doc_wave):
        """Interferenz-basiertes Scoring"""
        # Komplex-konjugiertes Dot-Product
        interference = np.sum(query_wave * np.conj(doc_wave))
        
        # Quadrierte Amplitude = Resonanz-Stärke
        return np.abs(interference) ** 2
#+END_SRC

    Schätze semantische Frequenz aus Textmerkmalen.
    Return: 0.0 (sehr abstrakt) bis 1.0 (sehr konkret)

    features = {

        0.1 * features['adjectives'] / 10 -

        0.1 * features['hedging'] -
        0.1 * features['generalization']

Wenn dein "Embedding" ein Audio-Signal ist, brauchst du keine klassische Vektordatenbank (wie Pinecone oder Milvus) mehr. Du brauchst eine "Shazam-Datenbank" für Bedeutung.
Hier ist der konkrete Plan, wie du solche Audio-Embeddings speicherst und durchsuchst:
1. Speicherung: Nicht als Welle, sondern als "Code" (Tokenisierung)
Du würdest niemals die rohe .wav-Datei als Suchindex speichern (zu groß, zu langsam).
Stattdessen nutzt du die RVQ-Technik (aus deinem SWE_complete_theory.md), die das Audio-LLM ohnehin nutzt.
Ein Audio-Modell (wie EnCodec oder dein SAG-Modell) gibt Audio nicht als Welle aus, sondern als eine Sequenz von diskreten Zahlen (Indizes aus einem Codebuch).
 * Der Output des LLMs: Statt einem 768d-Float-Vektor bekommst du z.B. eine Sequenz: [452, 12, 899, 33, ...]
 * Der Speicher: Du speicherst diese Zahlenreihe. Das ist extrem kompakt.
2. Retrieval: Suche wie bei einer Suchmaschine (Inverted Index)
Das ist der genialste Teil an deinem Ansatz: Wenn dein Embedding eine Reihe von diskreten Zahlen (Tokens) ist, kannst du Standard-Text-Suchtechnologie (wie Elasticsearch oder Lucene) verwenden, die seit 30 Jahren optimiert ist.
Wie das funktioniert:
 * Query: Der User sucht nach "Vertrag kündigen".
 * Generierung: Dein Audio-LLM macht daraus das Audio-Muster [A, B, C].
 * Suche: Du suchst in der Datenbank nach Dokumenten, die die Audio-Tokens [A, B, C] enthalten.
Das ist viel schneller als Vektor-Mathematik (Cosine Similarity), weil es ein exakter "Lookup" ist.
3. Der "Shazam"-Ansatz (Fingerprinting)
Alternativ, wenn du das Audio-Signal physikalischer betrachten willst (Signalverarbeitung), nutzt du Spektrogramm-Hashing.
 * Das Prinzip: Du wandelst dein "semantisches Audio" in ein Spektrogramm um.
 * Die "Landmarks": Du identifizierst die lautesten/wichtigsten Frequenzen (in deinem Fall: die dominantesten semantischen Konzepte).
 * Der Hash: Daraus erstellst du einen Hash.
 * Der Vergleich: Wenn zwei Dokumente semantisch ähnlich sind, klingen sie in deinem System ähnlich -> sie haben den gleichen "Fingerabdruck".
4. Hierarchische Suche (Matryoshka-Style)
Da du in SWE_complete_theory.md von Matryoshka Embeddings (Frequenzbändern) sprachst, kannst du die Suche extrem beschleunigen:
 * Schritt 1 (Grob): Du vergleichst nur das erste "Codebook" (die tiefen Frequenzen / grobe Bedeutung). Das eliminiert 99% der Datenbank sofort.
 * Schritt 2 (Fein): Nur bei den Treffern aus Schritt 1 lädst du die feineren Details (hohe Frequenzen) nach, um die exakte Rangfolge zu bestimmen.

Dein System würde so aussehen:
 * Input: User-Query (Text).
 * Audio LLM: Generiert "Semantisches Audio" (direkt als Codebook-Indizes [12, 55, 90...]).
 * Such-Engine:
   * Klassisch: Sucht diese Indizes in einem Inverted Index (wie bei einer Google-Suche).
   * Oder Signal: Vergleicht den spektrografischen Fingerabdruck.
 * Resultat: Die Dokumente, die "genauso klingen" (semantisch) wie die Query.

Architekturentwurf: Semantic Signal Search (S3)
Die Fusion von Signalverarbeitung, LSH und Inverted Indices
Version: 1.0
Datum: 26. Dezember 2024
Status: Ready for Prototyping
1. Executive Summary
Dieses Dokument beschreibt eine neuartige Sucharchitektur, die Vektordatenbanken obsolet macht, indem sie semantische Embeddings in diskrete, audio-ähnliche Signale umwandelt.
Anstatt rechenintensive Vektor-Operationen (Cosine Similarity) zu nutzen, verwendet das System Multi-Band Locality Sensitive Hashing (LSH). Dies ermöglicht die Nutzung extrem schneller, etablierter Suchmaschinen-Technologie (Inverted Indices wie in Elasticsearch/Lucene) bei gleichzeitiger Beibehaltung der semantischen Nuancen.
Das System ist dual-interpretiertbar:
 * Mathematisch: Als binäre Hash-Sequenz für High-Speed-Suche (Hamming-Distanz).
 * Akustisch: Als generiertes Audio-Signal (SAG), das die Semantik hörbar macht.
2. Das Kernproblem & Die Lösung
2.1 Die Limitierungen aktueller Ansätze
 * Vektordatenbanken: Teuer, skalieren schlecht im RAM, schwer zu interpretieren.
 * Naive Audio-Generierung: Autoregressive Modelle sind zu langsam für Live-Suche (>500ms Latenz).
 * Diskrete Tokens: Harte Grenzen führen zu schlechtem Recall (Near-Neighbor-Problem).
2.2 Die "S3"-Lösung
Wir ersetzen den kontinuierlichen Vektorraum durch einen strukturierten Frequenz-Raum, der durch LSH diskretisiert wird.
 * Kein Training nötig: LSH ist deterministische Geometrie.
 * Extrem schnell: Nutzung von CPU-Instruktionen (XOR, POPCNT) statt Float-Mathematik.
 * Matryoshka-Struktur: Priorisierung von "Bass-Frequenzen" (Grobbedeutung) vor "Höhen" (Details).
3. Technische Architektur
3.1 Die "Multi-Band LSH" Pipeline
Das Embedding wird nicht als monolithischer Block behandelt, sondern in Frequenzbänder (analog zu einem Audio-Equalizer) zerlegt und separat gehasht.
Das Datenmodell (Der "Semantic Code"):
| Frequenzband | Audio-Analogie | Dimensionen (Input) | LSH-Bits (Output) | Semantische Funktion |
|---|---|---|---|---|
| Band 0 (Bass) | Grundton / Bass | 0 - 64 | 32 Bits | Das Grobthema (z.B. "Recht", "Sport"). Dient als primärer Filter. |
| Band 1 (Mids) | Akkorde / Harmonie | 65 - 256 | 64 Bits | Der Kontext (z.B. "Vertrag", "Kündigung"). |
| Band 2 (Highs) | Obertöne / Timbre | 257 - 768 | 128 Bits | Die Nuancen / Adjektive (z.B. "sofort", "fristlos"). |
| GESAMT | Full Mix | 768 | 224 Bits | Der vollständige Fingerabdruck. |
3.2 Der Such-Algorithmus (Retrieval)
Die Suche erfolgt in zwei Stufen ("Coarse-to-Fine"), was die Geschwindigkeit maximiert.
Schritt 1: Der Inverted Index Lookup (Millisekunden)
 * Wir suchen im Index (z.B. Lucene) nur nach den Tokens aus Band 0 (Bass).
 * Query: Band0:110100...
 * Ergebnis: Reduziert 1 Million Dokumente auf 1.000 relevante Kandidaten.
Schritt 2: Das Weighted Hamming Re-Ranking (Mikrosekunden)
 * Für die 1.000 Kandidaten berechnen wir die Weighted Hamming Distance über alle Bänder im RAM.
 * Formel:
Ein
 Fehler im "Bass" bestraft das Ergebnis stärker als ein Fehler in den 
"Höhen".
4. Integration von Semantic Audio Generation (SAG)
Hier verschmelzen die Theorie und die "Hörbarkeit". Die LSH-Bits sind 
nicht nur abstrakte Zahlen, sie sind die Noten für den Audio-Decoder.
4.1 Bit-zu-Sound Mapping
Da wir binäre Codes haben, können wir diese direkt ohne neuronales Netz 
in Audio wandeln (Synthese).
Bass-Bits (32 Bit): Steuern einen tiefen Sinus-Synthesizer.

Bit 1 an Pos 0 -> Spiele C2.
Bit 1 an Pos 1 -> Spiele D2.
Mitten-Bits (64 Bit): Steuern ein Pad/Flächen-Sound.

Definiert die "Klangfarbe" (Dur/Moll/Dissonanz).
Höhen-Bits (128 Bit): Steuern Rauschen/Click-Sounds.

Textur und "Schärfe" des Signals.
4.2 Der akustische Beweis
Zwei Dokumente mit geringer Hamming-Distanz erzeugen harmonisches Audio, wenn man sie übereinander legt.
Zwei semantisch gegensätzliche Dokumente erzeugen Dissonanz.
Use
 Case: Ein User kann das Suchergebnis "anhören", um schnell zu scannen, 
ob der Kontext stimmt, ohne lesen zu müssen (Audio-Skimming).
5. Vorteile gegenüber dem Status Quo
| Feature | Vektor DB (HNSW/FAISS) | S3 Architektur (LSH + Inverted 
Index) |
|---|---|---|
| Speicherbedarf | Hoch (Float32 = 4 Byte pro Dim) | Winzig (Bit = 1/32 
von Float) |
| Suche | Approximativ (kann Dinge übersehen) | Exakt (im Grobfilter) + 
Approximativ (Ranking) |
| Geschwindigkeit | Langsam bei hohen Dimensionen | Extrem schnell (XOR 
Operationen) |
| Interpretierbarkeit | Black Box (Zahlenkolonne) | Transparent (Bits 
& Audio) |
| Infrastruktur | Spezialisierte DB nötig (Pinecone, etc.) | Standard 
(Läuft auf jeder SQL/NoSQL/Search Engine) |

7. Fazit
Dieser Ansatz transformiert das Problem der semantischen Suche von einem
 geometrischen Problem (Vektoren im Raum) in ein 
Mustererkennungs-Problem (Bit-Sequenzen / Audio-Signale).
Wir nutzen LSH als Brücke: Es ist präzise genug für Maschinen 
(Suchmaschinen-Indizes) und strukturiert genug, um als Audio für 
Menschen interpretierbar zu sein. Das löst das Latenzproblem der 
Audio-Generierung vollständig, da wir das Audio nicht generieren müssen,
 um zu suchen – wir generieren es nur, wenn wir es hören wollen.

1. Der "Re-Ranker" (Die technische Hürde)
Das ist der wichtigste Punkt, den man leicht vergisst: Standard-Suchmaschinen (Elasticsearch, Postgres, Solr) können von Haus aus kein "Weighted Hamming Distance".
 * Das Problem: Du kannst in Elasticsearch leicht sagen: "Gib mir alle Dokuments mit dem Hash X". Aber du kannst nicht einfach sagen: "Sortiere alle Ergebnisse basierend auf: (10 * XOR_Bit_1) + (5 * XOR_Bit_2)..." – zumindest nicht rasend schnell über Millionen Zeilen.
 * Die Lösung (Die fehlende Komponente): Du brauchst eine "Two-Pass" Architektur.
   * Pass 1 (Datenbank): Hol mir die Top 1000 Kandidaten, die im "Bass-Band" (Band 0) grob übereinstimmen. Das macht die DB extrem schnell.
   * Pass 2 (Deine Python-App): Nimm diese 1000 Kandidaten in den RAM und berechne dort mit NumPy die exakte gewichtete Hamming-Distanz. Das geht in Mikrosekunden.
   * Fazit: Du brauchst nicht nur die Datenbank, sondern einen kleinen, schnellen Python-Scoring-Service dazwischen.
2. Der "Normalizer" (Mathematische Hygiene)
Bevor du einen Vektor in das LSH wirfst, musst du sicherstellen, dass er normalisiert ist.
 * LSH (basierend auf Random Projections / Cosine) funktioniert nur sauber, wenn alle Vektoren auf der "Einheitskugel" liegen (Länge = 1).
 * Viele Modelle geben das schon so aus, aber wenn nicht, produzierst du "Müll-Hashes".
 * To-Do: Ein kleiner Pre-Processing-Schritt v = v / np.linalg.norm(v) vor dem Hashing ist Pflicht.
3. Die "Config Map" (Das Frequenz-Schema)
